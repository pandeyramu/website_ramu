How I Built a Typing Speed Test from Scratch

By Ramu Pandey · Published: 2026-05-23 · 6 min read

Building a typing speed test is a rewarding project for practicing DOM events, timing, and UX design. I built a simple, reliable test with precise timing, character-level feedback, and a scoring model that balances speed and accuracy. This article covers the design choices and implementation tips I used to create a responsive typing experience that works well on both desktop and mobile.

Goals and constraints

The test should begin on first keypress, measure elapsed time precisely, show per-character correctness, and provide a final WPM (words per minute) and accuracy percentage. It must be resilient to focus-loss and work with varied text sources. Latency and input lag are important factors to consider for fair scoring.

Text source and formatting

I used short passages stored server-side and delivered as JSON to the client. Each passage is split into characters and rendered in spans so that the UI can highlight correct and incorrect keystrokes in real-time. Breaking text into characters simplifies caret handling and visual feedback.

Input handling and timing

Reliable timing is key. I start the timer on the first valid keypress (not on focus) and use the high-resolution performance.now() API to compute elapsed milliseconds. Every keystroke compares the expected character and updates the DOM span's class to show correct, incorrect, or current caret position.

Scoring and accuracy

WPM is calculated as (correctChars / 5) / (minutes). Accuracy is (correctChars / totalTypedChars) * 100. To discourage rapid but inaccurate typing, the final score is reported alongside accuracy and a composite score that penalises typos more heavily. Users see per-character feedback and a summary at the end.

Handling corrections

Backspace behaviour needs careful handling. When users hit backspace, the test should move the caret back and allow correction, updating correctness counts accordingly. I implemented a cursor index and only allow backspace within typed range; deleting before start is ignored.

Mobile considerations

Mobile keyboards behave differently: autocorrect and suggestions can interfere with character-level testing. To mitigate this, the mobile UI uses a dedicated input element with autocorrect disabled and captures input events thoughtfully. The on-screen keyboard still provides a good enough experience for casual users.

Latency and fairness

Network latency should not affect timing. All timing runs client-side. However, initial passage fetch must be fast, so prefetch passages on page load to avoid delays. For multiplayer or competitive modes, use synchronized clocks or server-side validation to ensure fairness.

Accessibility and UX

Make sure visual cues are accessible: use color plus icons to indicate correctness, provide a visible caret, and allow keyboard-only operation. Offer adjustable font sizes and themes to improve readability for different users.

Testing and analytics

Collect simple, GDPR-friendly analytics: average WPM, difficulty of passages, and error hotspots. These signals help refine the passage pool and identify problematic character sequences. For testing, run unit tests for timing functions and end-to-end tests to simulate typing at different speeds and error rates.

Deployment tips

Serve the app as static files or integrate into a small backend. Compress assets, cache passages, and use a CDN for global users. HTTPS is required for secure usage and to prevent mixed-content warnings if integrating with other services.

Conclusion

A typing test is an excellent project that touches many parts of web development: precise timing, DOM manipulation, UX, and analytics. Start small, focus on reliability, and iterate with user feedback to refine scoring and passage selection. Over time, you can add leaderboards, user profiles, and learning-oriented features like targeted drills for common mistakes.
