How I Built a Typing Speed Test from Scratch

By Ramu Pandey Published May 2026 6 min read

Building a typing speed test is a rewarding project for practising DOM events, timing, and UX design. I built a simple, reliable test with precise timing, character-level feedback, and a scoring model that balances speed and accuracy across both desktop and mobile.

Goals and constraints

The test should begin on first keypress, measure elapsed time precisely, show per-character correctness, and provide a final WPM and accuracy percentage. It must be resilient to focus-loss and work with varied text sources. Latency and input lag are important factors to consider for fair and consistent scoring across different devices.

Text source and formatting

I used short passages stored as a JavaScript array and delivered directly to the client. Each passage is split into characters and rendered in spans so that the UI can highlight correct and incorrect keystrokes in real time. Breaking text into characters simplifies caret handling and makes the visual feedback much cleaner to implement.

Input handling and timing

Reliable timing is the core of the project. I start the timer on the first valid keypress rather than on focus, and use the high-resolution performance.now() API to compute elapsed milliseconds accurately. Every keystroke compares the expected character and updates the corresponding span's class to reflect correct, incorrect, or current caret position instantly.

Scoring and accuracy

WPM is calculated as correct characters divided by five, divided by the elapsed minutes. Accuracy is correct characters divided by total typed characters multiplied by one hundred. To discourage rapid but careless typing, the final score is shown alongside accuracy so users can see the full picture rather than just raw speed.

Handling corrections

Backspace behaviour needs careful thought. When users press backspace, the test moves the caret back and allows correction, updating correctness counts accordingly. I implemented a cursor index that tracks position and only allows backspace within the already-typed range. Attempting to delete before the start position is simply ignored to keep the state consistent.

Mobile considerations

Mobile keyboards behave differently from desktop ones. Autocorrect and word suggestions can interfere with character-level testing in unexpected ways. To reduce this, the mobile UI uses a dedicated input element with autocorrect and autocapitalize disabled. The experience is still usable on phones for casual practice even if competitive timing is better suited to a physical keyboard.

Accessibility and UX

Visual cues must be accessible to all users. I use colour combined with clear positioning to indicate correctness, provide a visible blinking caret, and ensure keyboard-only operation works throughout the test. Offering readable font sizes and sufficient contrast makes the tool usable for students with different visual needs.

Testing and analytics

Simple analytics help improve the tool over time. Tracking average WPM, common error positions, and which passages cause the most mistakes helps refine the passage pool. For code quality, unit tests cover the timing functions and scoring logic, and manual end-to-end tests simulate typing at different speeds and error rates to catch edge cases.

Deployment tips

The app is served entirely as static files with no backend dependency, which keeps hosting simple and fast. Compressed assets, cached passages, and a CDN for global users all contribute to a smooth experience. HTTPS is required to prevent mixed-content warnings and to keep the tool trustworthy for users sharing their results.

What I learned

This project taught me that browser timing APIs are more reliable than I expected and that character-level DOM updates can be fast if kept simple. The biggest lesson was that UX decisions like when to start the timer and how to handle backspace matter far more to the user than any visual styling choice. A tool that feels fair and responsive earns trust; one that feels inconsistent does not.
